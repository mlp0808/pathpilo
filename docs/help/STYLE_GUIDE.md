# Knowledge Base Style Guide

This guide is for AI-generated help content intended for WordPress publishing.

## Audience

- Primary: Admins and dispatchers
- Secondary: Team members
- Assume limited technical background

## Voice + Tone

- Clear, direct, and calm
- Practical over promotional
- Explain actions, outcomes, and troubleshooting
- Avoid buzzwords and vague claims

## Article Structure (Required)

1. **Title** (task-based, e.g. "How to optimize a day route")
2. **Who this is for**
3. **Before you start** (prerequisites)
4. **Steps** (numbered)
5. **What happens next** (expected result)
6. **Troubleshooting** (2-5 common issues)
7. **Related articles** (3-5 links/placeholders)

## Formatting Rules

- Use short paragraphs (1-3 lines)
- Use numbered steps for actions
- Use bullets for options/checklists
- Keep headings predictable:
  - `## Who this is for`
  - `## Before you start`
  - `## Steps`
  - `## What happens next`
  - `## Troubleshooting`
  - `## Related articles`

## Writing Constraints

- Do not invent functionality
- Do not reference hidden/internal implementation details
- Do not claim guaranteed outcomes (e.g. "always saves 30% time")
- If uncertain, include:
  - `## Needs confirmation`
  - Bullet list of unknowns

## Screenshot Policy

- Add placeholders where visuals help:
  - `[Screenshot: Route panel with Optimize button]`
- Prefer one screenshot per major step block, not every line

## Localization + Language

- Use consistent spelling variant across all articles (UK or US)
- Keep term usage aligned with `GLOSSARY.md`

## SEO Basics for Help Articles

- Put the exact user task in the title
- Include likely search phrasing in first 2 paragraphs
- Keep slug clean and task-based
  - Example: `optimize-day-route`

## Definition of Done

- Flow is accurate against product behavior
- Terms match glossary
- Troubleshooting is actionable
- Related links are included
- No speculative statements
